構造監査結果:

1. アーキテクチャの構造的問題:
現在、2つの独立したエージェントシステムが並存している:
・api/research_agents.py — 9エージェント（Python/FastAPI、Supabase直結）
・src/app/api/agents/stream/route.ts — 14エージェント（TypeScript/Next.js API Route）
これは二重投資。統合すべき。

提案: 会議エンジン（14エージェント）もFastAPI側に移行し、 Next.js API RouteはFastAPIへのプロキシのみにする。これにより:
・エージェント定義の一元管理
・Supabaseへの直接保存（会議結果を即座にDB書き込み）
・HF SpacesのGPU利用が将来的に可能（ローカルLLM実行）

2. 認知バイアス検出:
・アンカリング: 「既存コードを活かす」前提が強すぎる。 SSEストリーミングをVercel API Routeに残す判断は、Vercelの サーバレスタイムアウト（60秒 Hobby / 300秒 Pro）を考慮していない。 14エージェント × 6フェーズの処理時間が300秒を超える可能性あり。
・確証バイアス: 「全て無料枠で十分」は楽観的すぎる。 組織内で複数人が同時利用する場合のスケーリングを未検討。

3. 外部知見からの提案:
・CrewAI Studio: エージェント実行のObservabilityに LangSmith/Langfuse的なトレーシングを導入すべき。 各エージェントの入出力をstructured loggingとして保存。
・AutoGen v0.4: メッセージパッシングアーキテクチャの採用。 現在の「Phase順次実行」は単純だが、将来的にはエージェント間の 非同期メッセージングにすべき（例: ガードンがL1実行中にリアルタイムで警告を出す）。
・LangGraph: ステートマシンベースのワークフロー定義。 Phase遷移をグラフ構造で定義すると、条件分岐（例: リスクHighならPhase追加）が容易。
